GA4 BigQuery 电商用户行为分析

基于公开 GA4 数据集的 SQL 漏斗、渠道归因与用户生命周期分析
项目背景：将实习中 GA / Mixpanel 周报取数工作，向上扩展为"懂业务指标"，向下扩展为"能用 SQL 查询原始事件数据"

项目背景

实习期间负责 www.easesourcing.com 的 SEO 运营与数据周报工作：从 GA 取数、Mixpanel 漏斗看板每日记录、整理用户转化周报。在实际操作中发现——GA 界面看板是加工后的聚合数据，无法灵活拆解真实业务漏斗（涉及 GA 流量 + 前端点击事件 + 后端 Agent 行为三层合并）。

因此，使用 Google 公开的 bigquery-public-data.ga4_obfuscated_sample_ecommerce 数据集，用 SQL 复现一套标准用户行为分析流程，证明从运营执行到数据分析的能力迁移。

SQLBigQueryGA4用户漏斗渠道归因数据分析

分析方法 · 5 段 SQL 核心查询

1. 每日活跃用户（DAU）

基础聚合：GROUP BY + COUNT(DISTINCT)

SELECT event_date, COUNT(DISTINCT user_pseudo_id) AS dau
FROM ga4_obfuscated_sample_ecommerce.events_*
WHERE _TABLE_SUFFIX BETWEEN '20201101' AND '20201130'
GROUP BY event_date ORDER BY event_date;

2. 渠道来源拆解（归一化分组）

进阶技巧：CTE (WITH) + CASE WHEN 渠道归类 + SAFE_DIVIDE 百分比计算

WITH sessions AS (
  SELECT user_pseudo_id,
    LOWER(COALESCE(traffic_source.source, '(not set)')) AS source,
    LOWER(COALESCE(traffic_source.medium, '(not set)')) AS medium
  FROM ga4_obfuscated_sample_ecommerce.events_*
  WHERE event_name = 'session_start'
),
classified AS (
  SELECT user_pseudo_id, source, medium,
    CASE
      WHEN source = '(direct)' OR medium IN ('(none)', 'direct') THEN 'Direct'
      WHEN medium IN ('organic', 'seo') THEN 'Organic Search'
      WHEN medium IN ('cpc', 'ppc') THEN 'Paid Search'
      WHEN medium = 'referral' THEN 'Referral'
      ELSE 'Other / Unknown'
    END AS channel_group
  FROM sessions
)
SELECT channel_group,
  COUNT(DISTINCT user_pseudo_id) AS users,
  ROUND(100 * SAFE_DIVIDE(COUNT(DISTINCT user_pseudo_id),
    SUM(COUNT(DISTINCT user_pseudo_id)) OVER()), 2) AS user_share_pct
FROM classified GROUP BY channel_group ORDER BY users DESC;

3. 转化漏斗 + 窗口函数

进阶技巧：CASE WHEN 排序 + LAG() 窗口函数计算上一步转化率

WITH funnel_data AS (
  SELECT event_name,
    COUNT(DISTINCT user_pseudo_id) AS unique_users,
    CASE event_name
      WHEN 'view_item' THEN 1
      WHEN 'add_to_cart' THEN 2
      WHEN 'begin_checkout' THEN 3
      WHEN 'purchase' THEN 4
    END AS step_order
  FROM ga4_obfuscated_sample_ecommerce.events_*
  WHERE event_name IN ('view_item', 'add_to_cart',
    'begin_checkout', 'purchase')
  GROUP BY event_name
)
SELECT event_name, unique_users,
  ROUND(unique_users / LAG(unique_users)
    OVER (ORDER BY step_order) * 100, 2) AS conversion_rate_prev_step
FROM funnel_data ORDER BY step_order;

4. 新老用户分层

进阶技巧：CTE + JOIN + 首次访问日期判定

WITH first_visit AS (
  SELECT user_pseudo_id, MIN(event_date) AS first_date
  FROM ga4_obfuscated_sample_ecommerce.events_*
  GROUP BY user_pseudo_id
)
SELECT event_date,
  COUNT(DISTINCT CASE WHEN event_date = first_date
    THEN e.user_pseudo_id END) AS new_users,
  COUNT(DISTINCT CASE WHEN event_date > first_date
    THEN e.user_pseudo_id END) AS returning_users
FROM ga4_obfuscated_sample_ecommerce.events_* e
JOIN first_visit f USING(user_pseudo_id)
GROUP BY event_date ORDER BY event_date;

5. UNNEST 嵌套参数提取

进阶技巧：UNNEST(event_params) + 子查询提取自定义字段

SELECT event_date,
  (SELECT value.string_value FROM UNNEST(event_params)
    WHERE key = 'page_location') AS page_url,
  COUNT(*) AS pageviews
FROM ga4_obfuscated_sample_ecommerce.events_*
WHERE event_name = 'page_view'
GROUP BY 1, 2 ORDER BY pageviews DESC LIMIT 20;

业务理解与能力迁移

💡 核心洞察

GA 界面提供的是加工后的聚合看板，而 BigQuery 存储的是原始事件数据。实际业务中的用户漏斗往往跨越多个系统——如 easesourcing.com 的真实漏斗由三层组成：

1. GA 独立访客（流量层）→ 2. 前端"Try for Free"按钮点击（意向层）→ 3. 后端 Agent 开启新对话（转化层）

因此，完整分析需要将 GA 流量数据、前端埋点事件、后端业务数据合并，这正是本 SQL 项目所复现的分析思路。

🎯 能力证明

从运营执行（周报取数、看板记录）→ 向上理解业务指标含义，向下掌握 SQL 查询原始数据。将实习中的 GA / Mixpanel 实际场景与标准 SQL 分析能力结合，完成从运营到数据分析的技能迁移。
